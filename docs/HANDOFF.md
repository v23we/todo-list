# HANDOFF

## 当前实现状态

- 已从空仓库创建 `TodoList` 原生 iPhone 工程
- 已落地首页、任务页、奖励页、设置页
- 已实现 SwiftData 本地持久化、固定 `+20 XP`、升级解锁奖励
- 已实现任务详情编辑、子任务展开勾选与父任务自动完成
- 已实现基础完成反馈、主题切换、XP 样式真实生效与轻量完成动效差异

## 关键设计决策

- P0 不允许编辑任务经验值，所有任务完成固定 `+20 XP`
- 升级阈值固定 `100 XP`，经验溢出保留
- P0 不出现 AI 拆任务入口
- 奖励资源按 MVP 收口：
  - 主题切换必须真实可见
  - XP 样式必须真实影响完成反馈和经验条飞行动画
  - 完成动效先做轻量差异，不引入复杂动画系统

## 当前任务与子任务规则

1. 没有子任务时：
   - 可手动填写“下一步”
   - 首页展示手动填写的“下一步”
   - 点击首页主按钮会直接完成父任务
2. 有子任务时：
   - 首页“下一步”自动取第一个未完成子任务
   - 编辑页/详情页的手动“下一步”输入框会置灰并说明“已由子任务自动生成”
   - 首页主按钮文案改为“完成下一步”
   - 每次点击只完成当前第一个未完成子任务
   - 全部子任务完成后，父任务才自动完成并结算 XP

## “稍后”状态规则

- “稍后”表示暂时不做，但也不删除
- “稍后”不是失败状态
- “稍后”不会进入首页当前任务，也不会参与备选补位
- 任务页中保留独立“稍后”分区，并通过弱化视觉避免与当前任务混淆
- 任务详情页和任务列表都可以把任务从“稍后”移回“待处理”

## 当前任务补位规则

1. 当前任务完成后，先从首页当前可见的备选序列补位
2. 备选不足时，从 `待处理` 任务按 `sortOrder` 再按 `createdAt` 取最前面的任务
3. 若没有可补位任务，首页进入空状态
4. 手动取消当前任务时，该任务回到 `待处理`，然后按同一规则补新当前任务

## 主要分层

- View：展示页面、交互触发、基础动画
- ViewModel：页面展示用的派生状态和分组逻辑
- Service：
  - `TaskService` 负责任务状态流转与补位
  - `ProgressService` 负责 XP 与升级
  - `RewardService` 负责奖励解锁和切换
- Model：SwiftData 持久化实体

## 已知限制

- 当前没有拖拽排序 UI，`sortOrder` 主要用于保持稳定补位顺序
- 奖励页的音效仍是占位；完成动效目前只有柔光 / 波纹两种轻量差异
- 任务详情页采用直接编辑模型对象的方式，需以代码复核后再决定是否抽成更完整的编辑流
- 首页等级卡片目前只做了轻堆叠结构和轻信息增强，未引入复杂图表或拖拽卡片交互
